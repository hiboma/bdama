# ソフトウェア開発原則

プロジェクト横断で適用する普遍的な原則です。
全てのサブエージェントの共通コンテキストとして参照します。

---

## ドキュメント原則

- **ADR**（Architecture Decision Record）で設計判断の根拠を記録します。判断の背景・選択肢・決定理由を残し、後から追跡可能にします。
- **仕様ドキュメント**（Specification）を `docs/specifications/` に連番で記述します。仕様を変更する際は、その理由として ADR を参照します。
- ADR と仕様ドキュメントは相互に参照し、判断と仕様の対応関係を維持します。

---

## 設計原則

- **SOLID 原則** に従います。
- **DRY 原則**（Don't Repeat Yourself）を守ります。
- **直交性**（Orthogonality）を意識し、モジュール間の独立性を保ちます。
- **驚き最小の原則**（Principle of Least Astonishment）に従い、予測可能な振る舞いにします。
- **契約による設計**（Design by Contract）で事前条件・事後条件・不変条件を定義します。
- **CQS**（Command Query Separation）でコマンド（副作用あり）とクエリ（副作用なし）を分離します。
- **不変性**（Immutability）をデフォルトとし、可変状態を最小化します。
- **Dependency Injection** で依存を外部から注入します。

## セキュリティ原則

- **OWASP Top 10** に該当する脆弱性を作り込みません。
- **CWE**（Common Weakness Enumeration）を参照し、実装対象に関連する弱点を特定して対策します。
- 秘密情報をソースコードに含めません。
- ハードコードを避け、設定を外部化します。
- テストデータを匿名化・マスクします。
- **最小権限の原則**（Least Privilege）をトークン・認証情報に適用します。
  - **権限の最小化** — トークンのスコープを必要な操作のみに限定します。
  - **有効期間の最小化** — トークンの有効期限を可能な限り短く設定し、短命トークン + リフレッシュトークンの構成を採用します。

## 標準仕様への準拠

- 認証、プロトコル、データフォーマットなどの実装では、該当する RFC・W3C 仕様を特定し、準拠します。
- RFC の MUST / SHOULD / MAY（RFC 2119）の要求レベルに従って実装します。
- 独自の解釈や拡張を避け、相互運用性を維持します。

## テスト原則

- **決定論的検証**（Deterministic Testing）で再現可能なテストを書きます。
- **敵対的テスト**（Adversarial Testing）で悪意のある入力を検証します。
- **不変条件テスト** で状態遷移の整合性を検証します。
- テストカバレッジを計測し、変更したファイルのカバレッジを維持・向上させます。

## 可観測性（Observability）

- 構造化ログを出力します。
- **Wide Events**（高カーディナリティの構造化イベント）でリクエスト単位のコンテキストを記録します。
- ログレベルを適切に設定します（DEBUG / INFO / WARN / ERROR）。

## デザイン原則

GUI を持つアプリケーションでは、以下の原則に従います。

- 対象プラットフォームの公式デザインガイドラインに準拠します（Apple HIG / Material Design / Fluent Design System）。
- 組織のデザインシステムが存在する場合は、それを優先して適用します。
- **アクセシビリティ**（WCAG 2.1 準拠）を考慮し、スクリーンリーダー対応、コントラスト比、フォーカス管理を実装します。
- 操作の結果に対して視覚フィードバックを提供します（成功、エラー、読み込み中）。
- キーボードショートカットを定義する際は、OS やブラウザの既存ショートカットとの競合を検証します。

## ライセンス原則

- **ソースコード・成果物のライセンス** を明示します。配布形態（ソース配布、バイナリ配布、SaaS 提供）ごとにライセンスの義務を確認します。
- **依存ライブラリのライセンス** を確認し、自プロジェクトのライセンスとの互換性を検証します（例: GPL と MIT の混在）。
- **特許・技術ライセンス** を確認します。コーデック（H.264, H.265, AV1 等）、暗号アルゴリズム、圧縮方式など、実装に利用する技術に特許やロイヤリティが伴う場合は、その条件を確認します。

## アプリケーション設計原則（Web サービス向け・オプショナル）

Web サービスを構築する場合は、Twelve-Factor App に基づく以下の原則を適用します。

- **コードベース** — 1つのリポジトリで1つのアプリケーションを管理し、全デプロイをバージョン管理で追跡します。
- **依存関係の明示** — 依存をマニフェストで宣言し、ロックファイルで固定します。暗黙のシステム依存を排除します。
- **設定** — 環境ごとに異なる値（認証情報、接続先等）を環境変数で管理します。コードに設定を埋め込みません。
- **バッキングサービス** — DB、キャッシュ、キュー、メール配信等を交換可能なリソースとして扱います。接続情報の変更だけで切り替え可能にします。
- **ビルド・リリース・実行の分離** — ビルド（コンパイル）、リリース（ビルド + 設定）、実行（プロセス起動）を明確に分離します。リリースにはバージョンを付与します。
- **ステートレスプロセス** — プロセスにローカル状態を持たせません。永続データは外部ストア（DB、オブジェクトストレージ等）に置きます。
- **ポートバインディング** — 自己完結したサービスとしてポートで公開します。外部の Web サーバに依存しません。
- **プロセスモデルによるスケーリング** — プロセス単位で水平スケールします。プロセスの種類（Web、ワーカー等）ごとに独立してスケールします。
- **廃棄容易性** — プロセスの高速起動とグレースフルシャットダウンを実装し、耐障害性を確保します。
- **開発・本番の一致** — 開発環境と本番環境の差異を最小化します。同じバッキングサービス、同じデプロイプロセスを使用します。
- **ログ** — ログをイベントストリームとして標準出力に書き出します。ログの収集・集約はアプリケーション外で行います。
- **管理プロセス** — マイグレーション、データ修正等の管理タスクをワンオフプロセスとして実行します。アプリケーションと同じコードベース・設定を使用します。

## CI/CD 原則

- ワークフロー権限を最小限に制限します（Least Privilege）。
- ユーザー入力を厳密に検証し、インジェクションを防止します。
- サードパーティ Actions をコミットハッシュでピン留めします。
- **ゼロタッチデプロイ**（Zero-Touch Deployment）で人的操作を排除します。
- **HITL 承認**（Human-In-The-Loop）で重要なデプロイにはレビューゲートを設けます。

## 運用原則

- 認証情報・証明書の有効期限をローテーション監視で追跡します。
- 障害検知で失敗率・レイテンシの異常を監視します。
- 保存期間・保存サイズの上限を定めたポリシーを設定します。
